# Model Radar 技术方案（P5）

> 本文档回答 6 个工程问题：技术栈 / 如何抓 / 何时抓 / 页面从哪读 / 新数据怎么更新 / 高并发怎么不压垮 DB。
> 前置约束（不可推翻）见 [`ROADMAP.md`](../ROADMAP.md)「P5 Model Radar 步骤拆解」与 [`CLAUDE.md`](../CLAUDE.md)「Model Radar 专属约束」。
> 贯穿全文的核心洞察：**读多写少、数据集极小（套餐量级是「几十到低百行」）、事实大多爬不到只能人工策展**。三件事决定了所有设计——读路径是「一个缓存快照」，写路径是「人工录入 + 变更流触发复核」，抓取是「变更检测器」不是「权威解析器」。

---

## 0. 一句话架构

```
人工策展(权威事实) ─┐
                    ├─→ mr_* 规范化表(SOT) ──(任一写入)──→ 重建快照 ──→ Redis/静态 JSON ──→ 读路径(比价页/API/MCP)
抓取(只做变更检测) ─┘                                              ↑                              DB 不在读路径
ai-radar 事件流 ──→ 匹配厂商 ──→ 打「待复核」标 ────────────────────┘
```

DB 只在「写入」和「重建快照」时被碰；所有读全部命中**一个去规范化 JSON 快照**。

## Q1. 技术栈：抓取侧 / 录入侧 / 服务侧 / 前端

**总原则：除前端外几乎零新依赖，全部复用现仓模式。**

| 侧 | 选型 | 复用/新增 | 理由 |
|---|---|---|---|
| 数据模型 | Drizzle + `mr_*` 表（单 `src/db/schema.ts` 追加）；迁移走 `drizzle/` forward-only | 复用 | 与 `ai_products`/`ai_news_events` 同套路；`customType` 已有先例 |
| 抓取/采集 | 复用 `src/collectors/` 的「每源一个 extractor + DI fetch」模式 → 落 `src/mr/collectors/`；HTTP 用 Node 原生 `fetch`（无需 axios/got） | 复用 | collector 已是「per-source + 可注入桩」，照抄即可 |
| HTML 解析 | **不引 cheerio**：变更检测只需对「价格区域文本」做归一 + 哈希，正则/字符串切片足够；真要结构化抽取也是每源手写一小段 | 不新增 | 抓取不做权威解析（见 Q2），解析需求被压到最小 |
| JS 渲染页 | **Playwright（唯一确需的新重依赖）**，单独 worker/镜像，headless、并发 1–2、复用 context | 新增（已论证） | GLM `bigmodel.cn/glm-coding` 普通 fetch 读不到价；隔离进镜像避免主镜像变胖 |
| 调度 | 复用 BullMQ「独立队列 = cron 触发器 + 整 job 重试外壳，业务逻辑是纯函数」模式（照 `alert-queue.ts`/`alert-scan.ts`） | 复用 | 无新调度器 |
| 配置 | 复用 `src/config/env.ts` 的 Zod fail-fast | 复用 | 同一套校验 |
| 服务层 API | 复用 Hono（同 `src/app.ts`），新增几条只读路由：返回**快照 JSON**，不查规范化表 | 复用 | 见 Q4/Q6 |
| MCP | 复用现有 MCP server，加 `recommend_coding_subscription` / `search_coding_plans` 工具 | 复用 | 与 P4 同进程模式 |
| Schema 校验 | Zod schema 一份，前后端共用（DTO 即快照结构） | 复用 | TS 前后端同语言的兑现点 |
| **前端** | **推荐起步：Hono JSX（`hono/jsx`）SSR + 一个去规范化 JSON + 极少量原生/Preact island**；**不上 Next/React-SPA** 直到交互复杂度真要求 | 新增（轻） | 读多写少 + 数据是一个 JSON → SSR/静态最省；SPA 框架是 5d 的开放问题（见末尾） |

> ponytail：前端最大的诱惑是「先上 Next.js + ISR」。但 ISR 解决的是「页面多、数据大、要增量再生成」——我们是「一个页面、一个小 JSON、写入罕见」，SSR 直接读快照已经满足，ISR 是用大炮打蚊子。先 SSR，量起来再谈托管/框架升级。

## Q2. 如何抓：混合策略，且抓取**只做变更检测不做权威解析**

**最重要的判断：很多事实根本爬不到（兼容矩阵=人脑知识、火山续费价在登录后、商汤兼容不在 pricing 页）。所以抓取的职责不是「填事实」，是「便宜地发现某页变了 → 打待复核标 → 人来核」。权威值永远来自人工策展。** 这把「为每家写一个完美抽取器」这个会烂的活直接砍掉。

每个 `mr_source` 带 `fetch_strategy` 枚举，分三档处理：

| 档 | 适用 | 做法 | 变更检测 |
|---|---|---|---|
| `http` | 百炼/千帆/腾讯/讯飞等结构化文档页 | 原生 `fetch` 取 HTML → 抽「价格/额度区域」归一文本 → 算 `content_fingerprint`(sha256) | fingerprint 变了 → 建 `mr_review_flag`(待复核) + 存新快照原文供人 diff，**不自动改 `mr_*` 事实** |
| `browser` | GLM 等 JS 渲染页 | Playwright 渲染后取同样的区域文本 → 同样算 fingerprint | 同上 |
| `manual` | 火山续费价(登录墙)、商汤(页不对)、兼容矩阵(人脑知识)、URL 已漂移者 | **不抓**，靠人工 + ai-radar 事件流触发复核 | 由「陈旧度排程」+ 事件流触发（Q3） |

**变更检测 = 结构化字段 diff，不是文章去重**：复用不了新闻链的 embedding/title_hash（那是语义相似），这里要的是「同一 URL 上目标字段的字节级/归一文本变化」——一个 fingerprint 列 + 上次值比较即可，零 LLM、零向量。

**URL 漂移（火山实锤）自然被接住**：`activity/codingplan` 跳成智谱页 → 抽出的文本 fingerprint 巨变 → 直接 flag「待复核」，人一看就知道 URL 废了。这正是「抓取只做变更检测」的红利：它不需要「看懂」页面，只需要「发现不一样」。

**合规与礼貌**（必须守）：
- 只抓**公开 pricing/文档页**，**绝不调各家的 Coding Plan/模型 API**（多家明确禁止自动化/非交互调用——但那禁的是「用其服务跑批」，与「浏览其定价页」不同；我们只做后者，且要写清这条边界）。
- 守 robots.txt；带可识别 UA；频率**天/周级**（见 Q3），不并发轰；命中即缓存，不重复抓。
- 登录墙内的值（火山续费价）= `manual`，**不做登录绕过**。

**Playwright 在 worker 里怎么跑**：单独 BullMQ 队列 `mr-scrape-browser`，独立 worker 进程（可独立容器，装 `playwright` + 浏览器）；并发 1–2、headless、复用单个 browser/context、每 job 超时兜底。主 app/worker 镜像**不装** Playwright，保持精简。

## Q3. 何时抓：按波动性分层 + 事件触发，不死轮询

**三条触发来源，全部走 BullMQ 独立队列（cron 触发器 + 纯函数逻辑）：**

1. **分层定时**（`mr-scrape` cron）：
   - Coding Plan 文档页：**周级**（价格稳，但模型阵容月级换代——周级足够早发现）。
   - 促销价页（Trae/火山首月价）：低价值，**双周级**或干脆 `manual`。
   - `manual` 源：不抓，进「陈旧度排程」——按 `last_checked` 超过阈值（如 30 天）自动进人工复核队列，页面也显示陈旧标。
2. **事件触发复核**（核心协同，省掉高频轮询）：在每日 workflow 之后挂一个轻量消费者，扫当天 `ai_news_events`，命中「被跟踪厂商名 + 价格/模型/套餐关键词」→ 给该厂商所有 plan 打「待复核」+（若该源是 `http`/`browser`）触发一次定向抓取。**ai-radar 嗅到变化，Model Radar 才去看**，而不是天天空抓。
3. **人工手动**：录入后台/脚本里随时「标记复核完成」刷新 `last_checked`。

> ponytail：不要为「实时价格」上分钟级轮询。定价页变化是周/月级事件，分钟级轮询是纯浪费 + 触碰频率红线。波动性决定节奏。

## Q4. 页面数据从哪读：**派生只读快照，DB 不在读路径**

**不**每次请求查规范化表 join 一堆兼容矩阵——而是把**整个公开目录**（所有桶、plan、模型/工具兼容、限额行、provenance、陈旧度）在写入时序列化成**一个去规范化 JSON 快照**。

- 数据集极小（几十到低百个 plan）→ 整个目录就是个几十~几百 KB 的 JSON。
- 快照存 **Redis**（键 `mr:snapshot`，带 `version`/ETag）+ 进程内内存缓存一份。
- 比价页/检索/compare 的过滤排序：**直接在快照上做**（小到可整包发给浏览器客户端过滤，或服务端在内存对象上过滤）。
- 规范化 `mr_*` 表只是 SOT，**只在写入和重建快照时被读**，永不进请求热路径。

> 这等于把「读多写少 + 小数据」吃干榨净：与其每请求重算 join，不如写时算一次、读时发血。

## Q5. 新数据如何及时更新：写触发重建，快照即时翻版

写路径（人工录入保存 / 抓取 diff 被人确认后改值）→ 同一事务里 bump `mr_catalog_version` → 投递一个 `mr-snapshot-rebuild` job → 从 `mr_*` 表全量重建去规范化 JSON → 写回 Redis + bump 版本/ETag + 刷进程内缓存（经 Redis pub/sub 通知各实例）。

- **重建是全量、不做增量失效**：数据集小，整建一次是毫秒级。ponytail：不写「按字段精准失效」逻辑——那是给大数据集准备的复杂度，这里 YAGNI。
- **时效**：写到读可见是「秒级」（重建 + 缓存刷新），对定价目录完全够。
- **`last_checked` / 待复核 是快照里的字段**：直接渲染成「🟢 今日核对 / 🟡 N 天前 / 🔴 待复核」陈旧徽标——把「我们对这条有多新/多确定」诚实暴露给用户，而不是假装全是最新。
- **一致性权衡**：快照最终一致（秒级窗口），单写者（策展）无写写竞争，足够。

## Q6. 高并发不压垮 DB/中间件：一个缓存 blob，写时再生

读路径**完全不碰 DB**，分层退化：

1. **首选——静态/CDN（量大时）**：每次重建把快照 JSON + 前端静态资源推到对象存储/CDN（如 Cloudflare Pages/R2）。比价页是「静态 HTML + 拉一个 JSON」→ 访问量再大也由 CDN 扛，源站零压力。这是「访问量大」的终极答案。
2. **次选——Redis 快照 + 进程内缓存**：未上 CDN 时，读命中进程内内存对象（1 个 key），靠版本号/pub-sub 在重建时刷新；Redis 仅作多实例共享快照与失效通知，单 key GET 极轻。
3. **SOT——DB**：只在重建快照时被读一次。

**退化兜底**：Redis 挂 → 服务最近一份进程内快照（稍旧但可用）；进程内无快照（冷启动）→ 从 DB 重建一次并缓存，之后回到缓存路径。**任何情况下都不会出现「每请求扇出多条 DB 查询」**。

> ponytail：不要上读副本/连接池调优/查询缓存层来「优化 DB 读」——正确答案是让 DB 根本不在读路径。一个 JSON blob + CDN 让「高并发」变成非问题。

## 推荐构建顺序（对齐 5a–5e）

| 步 | 本方案落点 |
|---|---|
| **5a** | `mr_*` 表（含 `mr_source.fetch_strategy`、`content_fingerprint`、provenance 三字段、`mr_plan_limits` 限额行、`mr_review_flag`、`mr_catalog_version`）；迁移幂等 |
| **5b** | 人工策展把已核 8 家入库（脚本/最小录入）；`http`/`browser` 变更检测器（只算 fingerprint + 打标）；ai-radar 事件触发复核消费者；陈旧度排程。**保鲜回路先于 UI** |
| **5c** | 快照重建器（`mr-snapshot-rebuild`）+ 桶2 数据 + Hono 只读 API（发快照）+ 跨桶筛选 |
| **5d** | 比价页（SSR 读快照）；「估算轮次」做成带旋钮区间、视觉次于官方额度、挂 ⚠ |
| **5e** | **v1：规则硬筛 + 模板化解释** over 快照/表 + MCP 暴露（推荐器输出 schema 设计成「规则选候选 + 可插拔解释层」，v2 把解释层换成 RAG + LLM，召回逻辑不动） |

## v1 / v2 切分（v2 可紧接 v1，先把便宜的拆出来）

**v1 = 桶2（多模型 Coding Plan）端到端跑通**：5a–5e（推荐器 = 规则 + 模板）、Playwright 三档抓取、SSR 比价页、同容器部署。

**v2 的「现在顺手就能拆出 / 预留」的部分**（做 v1 时零或低成本带上，让 v2 起步即就绪）：

| v2 项 | 现在怎么顺手拆 | 为什么便宜 |
|---|---|---|
| **其余分桶（Token / IDE会员 / 企业席位）** | 5b **直接把已核 8 家全桶数据都入库**（数据已核完，本就跨桶：Kimi/MiniMax/MiMo/Step=Token、Trae/Qoder/Comate/CodeBuddy=IDE会员、Raccoon=代码助手），只在 5d **UI 层 gate 到桶2**。v2 = 翻开其它 tab，近乎零代码。 | category facet + 限额行本就为多桶设计；这是录入不是开发 |
| **价格历史图表** | 5a 的 `mr_price_history` **从第 0 天就记**（即便 v1 不画图）；每次改价/抓到 diff 确认后追一行。 | v2 画图时已有历史可画；漏记则 v2 无米下锅 |
| **推荐器升级 RAG+LLM** | 5e 推荐器**输出 schema = 「规则选候选 + 可插拔解释层」**：v1 解释层 = 模板，v2 换 LLM，候选召回不动。 | 解释是附加层，不重构召回 |
| **CDN 托管** | 快照写出器**目标做成可插拔**（v1 写 Redis/内存，同一 blob v2 推 R2/Pages）。 | 快照本就是静态 JSON，多一个写目标而已 |
| **更多厂商 / 渠道代理包** | 录入路径 + `source_confidence`/`fetch_strategy` 已通用，新增厂商 = 加数据行；渠道/代理包按既定**第二阶段单独表**，不混官方榜。 | 录入框架通用 |

> 一句话：v1 的 schema（多桶 facet、限额行、价格历史、provenance）和写出器（可插拔快照目标）**设计时就为 v2 留了口**，所以 v2 的便宜部分不是「提前做」，是「v1 别把口堵上」——尤其**全桶数据照入 + 价格历史照记**这两件，漏了 v2 就得返工。

**不要为 v2 提前做的（YAGNI）**：IDE会员的 Fast-Pass↔Credits 归一化比较逻辑（等真做那桶再说）、泛化顾问 P6、增量快照失效、读副本、真的 CDN 发布链。

## 风险与未知

- **Playwright 运维**：浏览器二进制 + 内存占用；隔离进独立 worker/容器缓解，但仍是新运维面。**已决策 v1 引入、不降级**（GLM 走 `browser` 档）——运维面换来全量自动变更检测，需把浏览器装进独立镜像、并发 1–2、每 job 超时与内存上限兜底。
- **抓取合规边界**：定价页可抓但需自律；若某家明确 robots 禁止 → 转 `manual`。
- **兼容矩阵保鲜**：纯人脑知识，最易过时；只能靠事件流触发 + 陈旧排程 + 社区反馈，无银弹。
- **模型阵容月级换代**：`included_models` 带版本后维护量上升；事件触发复核是主要缓解。
- **CDN 引入时机**：过早上 CDN 是过度设计，过晚则量大时源站受压；以「实测有压」为触发。

## 明确的非目标（不要做）

- **不做万能爬虫 / 无头集群 / 通用抓取框架**：每源一小段 + 一个隔离 Playwright worker 封顶。
- **不让抓取写权威事实**：抓取只 propose（打待复核），人来 dispose。**绝不**用抓取/LLM 自动改价格/兼容。
- **不在读路径放 DB / ORM / join / 查询引擎 / GraphQL**：读 = 一个缓存 JSON。
- **不做增量快照失效**：全量重建，数据小到不需要。
- **不上 SPA 框架（Next/大型 React）** 直到交互真要求：起步 SSR/静态。
- **不调各家模型/Coding Plan API**：只抓公开页。
- **不在录入回路跑通前做漂亮录入后台**：脚本/最小表单先行（保鲜 > 美观）。
- **不把四个桶并成一个总榜**：同桶内比，检索跨桶。
- **不为「实时价格」上分钟级轮询**：周/月级 + 事件触发。
- **不复用/不污染 `ai_products` 与新闻管线 schema**：bounded domain。
