## 新增需求

### 需求:飞书自定义机器人通道推送

系统必须提供一个飞书推送通道，基于飞书自定义机器人 webhook（原生 fetch + 签名）把每日日报/产品发现/告警推送到配置的飞书群。**飞书是可选通道，不破坏 P1 纯 Telegram 部署的向后兼容**：未配置飞书（无 `FEISHU_WEBHOOK_URL` / `FEISHU_SIGN_SECRET`）时，飞书**不纳入「已配置通道」集**、不参与分发、不报错——「向所有已配置通道分发」（daily-intel-pipeline）据此只发已配置的通道（至少 Telegram）。仅当**配置不完整**（声明启用飞书但 webhook URL 或签名密钥缺其一）时，env 校验必须以明确错误快速失败，禁止用空值发送或静默半启用。消息必须渲染为飞书 JSON 卡片（区别于 Telegram 的 MarkdownV2），卡片内跳转只用按钮/文字链方式（飞书自定义机器人不支持点击回调到服务端，禁止设计依赖回调交互的卡片）。所有外部调用必须带重试与错误日志，发送失败禁止静默吞掉。

飞书自定义机器人限流为单租户单机器人 100 次/分钟、5 次/秒；系统调度推送时必须避开整点/半点（如 10:00、17:30）等高压时刻以降低 11232 限流错误概率。**「避整点」须有可验证判据**：每日/周报/告警等定时 cron 的分钟字段默认值 **∉ {0, 30}**（如 08:03），供 tasks 机械断言。P2 各推送路径（日报每日一条、产品/周报低频、告警高频但单条）天然稀疏，单条卡片发送远低于 5 次/秒，**P2 不实现发送侧节流闸**；但若未来同一秒并发多条卡片（如批量分条），必须补 5 次/秒发送节流——本期记为已知边界、不实现。

#### 场景:推送渲染为飞书 JSON 卡片
- **当** 飞书通道推送一份日报
- **那么** 消息以飞书 JSON 卡片格式发送、跳转用按钮/文字链，发送成功

#### 场景:未配置飞书时纯 Telegram 部署照常启动
- **当** P1 部署升级到 P2 但未配置任何飞书 env（webhook/secret 均空）
- **那么** 飞书不纳入「已配置通道」集、不报错，系统照常启动并只向 Telegram 分发，向后兼容

#### 场景:飞书配置不完整时启动即报错
- **当** 声明启用飞书但 webhook URL 或签名密钥仅配其一（不完整）
- **那么** 系统以明确错误信息快速失败，禁止用空值发送或静默半启用

#### 场景:定时触发避开整点降低限流
- **当** 检视飞书相关定时 cron 默认值的分钟字段
- **那么** 分钟字段 ∉ {0, 30}（如 08:03），机械可断言不落整点/半点，以降低 11232 限流错误概率

### 需求:飞书推送幂等按 channel 独立

系统必须以 `push_records` 的 `UNIQUE(target_type, target_id, channel, push_date)` 保障飞书通道推送幂等，飞书推送记录的 `channel` 必须为 `feishu`。**选题统一（Model B）+ 各通道可靠补发**：每日只选一份 channel-blind Top N（不按 channel 选题，见 daily-intel「Top N 组合分选择」），同一份发放给 Telegram 与飞书；**各通道经 dispatcher 在 dispatch 层做 per-channel 跨天幂等**——`computePendingSet` 排除「该 channel **任一 push_date** 已 success」+ `UNIQUE` 四元组兜底同日并发，故 Telegram 已 success 不抑制飞书发该事件（两通道各发一次），且飞书某次失败后跨次仍在飞书待发、可靠补发（不丢），飞书一旦 success 即不再重发。飞书推送必须沿用「先写 `pending` → 调 API → 成功 `success` / 失败 `failed` 保留错误信息可重试」，唯一键冲突即跳过。

#### 场景:同一份日报当日发放两通道各发一次
- **当** 当日一份统一 Top N 发放给 Telegram + 飞书，Telegram 已 success、飞书当日待发集合仍含该事件（如整 job 重试中）
- **那么** 该事件在飞书通道仍进入待发集合并被发送（per-channel 跨天幂等：飞书从未 success → 待发；不因 Telegram 已 success 而被抑制），两通道各发一次

#### 场景:飞书当天重跑不重复推送
- **当** 飞书通道当日日报已 success 后，在同一 `push_date`、`channel='feishu'` 再次执行
- **那么** 待发集合排除已 success 的事件后为空，不重复发送
