## 新增需求

### 需求:周报定时汇总与推送

系统必须提供一个独立于每日日报的周期性周报任务（BullMQ 周级 cron 触发），汇总过去一周的高价值事件/产品并按程序选择后推送到配置通道。**「过去一周」窗口边界必须与 `iso_week` / `push_date` 同源**：取 `[上一 ISO 周周一 00:00, 本 ISO 周周一 00:00)`（Asia/Shanghai），其中 **`本 ISO 周周一` = 触发时刻所在 ISO 周的周一 00:00（与触发 weekday 无关——无论 cron 配在周一、周三还是周日触发，`本周一` 恒为触发时刻当周的周一，窗口确定）**。**禁止用滚动 7×24h 窗口**——滚动窗口与 `iso_week` 锚定的周边界不一致，会使周一前后两次触发落在不同 `iso_week`（UNIQUE 不挡）却覆盖高度重叠的内容、把重复内容跨周各推一次。周报选择名单与排序必须由程序规则（组合分、时间窗口）决定，禁止由 LLM 决定最终名单；周报内每条的中文摘要复用已落库的 `summary_zh` / `headline_zh`，不重复触发逐条 LLM 调用。周报选入产品时必须**复用每日产品推送的 `merge_conflict` 排除谓词**（排除标记 `merge_conflict` 的 `product_id`），避免同一真实产品因散为多个 `product_id` 在周报正文内重复列出（展示一致性）。周报触发时刻必须避开整点/半点以降低飞书限流。所有外部推送调用必须带重试与错误日志。

#### 场景:每周定时汇总过去一周内容并推送
- **当** 周报 cron 到点触发
- **那么** 系统按程序规则从过去一周窗口选出高价值事件/产品，复用已落库摘要拼成周报并推送

#### 场景:周报名单由程序而非 LLM 决定
- **当** 确定哪些内容进入周报
- **那么** 名单与排序依据程序的组合分与时间窗口规则，禁止由 LLM 决定最终名单

### 需求:周报推送幂等按周粒度

系统必须以 `push_records` 唯一约束保障同一周的周报不重复推送。周报四元组必须为 `target_type='weekly'`、`target_id=iso_week`（如 `2026-W24`）、`channel`、`push_date=该 ISO 周周一（Asia/Shanghai）`。

**`target_id` 与 `push_date` 必须同源锚定「被汇总窗口」的 ISO 周，而非触发时刻所在周（消除歧义，绝不可省）**：`iso_week` 与 `push_date` 都取**被汇总窗口 `[上周一, 本周一)` 对应的那个 ISO 周**（即「刚结束的完整一周」）的标签与其周一日期——**不取触发时刻所在的 ISO 周**。否则触发时刻在周边界附近抖动（如周一 00:02 触发，触发周=W24 但汇总窗口=W23）会使 `target_id`（汇总周 W23）与 `push_date`（若误取触发周 W24 周一）**指向不同周、错配**，UNIQUE 四元组虽仍唯一但幂等语义漂移。「周」边界锚定 Asia/Shanghai 固定 weekday（周一），与日报 `push_date` 时区同源，防跨零点把一份周报算成两周。独立 `target_type='weekly'` 使周报与日报、产品发现、实时告警在 `push_records` 中互不挤占。

周报推送必须**复用 telegram-push/feishu-push 的同一套「待发→`pending`→原子送达→`success`/`failed`」状态机核心**（仅 `target_type` 与幂等键口径不同），禁止另写一套漂移的状态机；唯一键冲突即跳过。周报任务必须带**独立单例锁**（如 `weekly:{channel}:{iso_week}`）或 DB 原子 claim，防两并发实例各发一条；该锁须 job 级短时持有 + 完成/崩溃可靠释放（带 TTL 或 `finally`），锁键不含时间故释放语义不可省（同 telegram-push 单例锁要求）。

#### 场景:同一周周报不重复推送
- **当** 某周周报已 success 后，周报任务在同一 `iso_week`、同一锚定 `push_date` 再次执行
- **那么** 该周周报因 `UNIQUE(weekly, iso_week, channel, push_date)` 冲突被跳过，不重复推送

#### 场景:同一 ISO 周内抖动触发不改变 target_id/push_date
- **当** 周报任务在同一 ISO 周内的不同时刻抖动触发（如周一 09:07 与周一 09:10 两次，cron 默认周一早晨触发、真实抖动为秒/分钟级）
- **那么** 二者落在同一触发 ISO 周，其「被汇总窗口 `[上周一,本周一)`」相同，`target_id=iso_week` 与 `push_date` 完全一致、严格同源，UNIQUE 四元组一致、不因抖动重复推送（注：确定性非滚动规则下，跨 ISO 周边界的两次触发——如周日 23:59 与周一 00:02——本就属不同 ISO 周、对应不同的「刚结束一周」，锚定不同 `iso_week` 是有意的正确行为，非重复推送）

#### 场景:周报与日报互不挤占
- **当** 同期既推日报又推周报
- **那么** 二者因 `target_type` 不同各自独立幂等，互不影响彼此推送状态
